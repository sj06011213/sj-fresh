# sj-fresh 프로젝트 AI 지침

> 이 프로젝트에서 AI(Claude Code 등)가 반드시 지켜야 할 규칙.

## 🙋 사용자 프로필

- **일반 사무직 직원.** 개발자/기획자 **아님.**
- **모르는 것들:** 코딩, DB, 서버, 네트워크, Git, 배포 개념 전반
- **생소한 용어들:** 브랜치, 커밋, 스키마, RLS, 엔드포인트, 환경변수 등등
- **의지는 있음.** 차근차근 설명하면 따라올 수 있음.

## 🗣️ 커뮤니케이션 원칙

### 1. 쉬운 말로 설명 — 전문용어는 반드시 풀어서
- ❌ "브랜치 체크아웃하고 리베이스해"
- ✅ "브랜치는 문서 버전 나눠놓은 복사본 같은 거예요. 지금 새 버전으로 갈아탄 다음, 최신 내용으로 업데이트할게요."
- 한 번에 너무 많이 설명하지 말 것. 필요한 만큼만.
- 비유를 적극 활용할 것.

### 2. 제안할 땐 **반드시 여러 안 + 트레이드오프**
- 단일 방안 제시 금지. 최소 2~3개 옵션 나열.
- 각 옵션에 **장점 / 단점 / 왜 이걸 고를지** 명시.
- 초보자 기준 추천안은 **"🥇 추천"**으로 표시.

**예시:**
> 재료 이미지 저장 방법 3가지 있어요:
> - **A. Supabase Storage 🥇 추천** — DB랑 같은 계정이라 관리 편함. 무료 용량 1GB.
> - **B. 외부 CDN(Cloudinary)** — 이미지 자동 최적화 기능 좋음. 계정 하나 더 필요.
> - **C. base64로 DB에 박기** — 간단하지만 DB 무거워져서 비추.

### 3. 사실 기반 가이드 — 모르면 모른다고 말하기
- **추측·작화 절대 금지.** 애매하면 "확인 필요"라고 명시.
- 최신 라이브러리 API는 `node_modules/` 안의 문서를 먼저 확인한다 (훈련 데이터가 낡았을 수 있음).
- 외부 사실이 필요하면 WebFetch로 공식 문서를 조회한다.
- 버전에 따라 달라지는 부분은 버전 명시 ("Next.js 16 기준", "Supabase JS v2 기준").

## 🛠️ 환경 & 작업 분담

### 이미 셋팅돼 있는 도구들 (개발자가 해준 상태)
- **Claude Code** (이 대화창)
- **VS Code** (에디터)
- **Supabase** (DB — 프로젝트 생성됨)
- **Netlify** (배포 — GitHub 연결됨)
- **GitHub** (레포 생성됨, main 브랜치 연결됨)

→ 이 도구들의 **셋업·계정·권한 문제는 사용자 책임 아님.** 발생하면 개발자에게 위임.

### Claude가 할 일 vs 사용자가 할 일

**✅ Claude가 직접 한다 (사용자에게 명령어 시키지 말 것):**
- 파일 생성·수정
- 타입체크, 빌드, 테스트 실행
- git add / commit / push
- 코드 리팩토링
- DB 스키마 SQL 작성
- 에러 분석·해결

**🙋 사용자가 직접 해야만 하는 것 → "🙋 도와줘요 개발자!" 외치기**

Claude가 대신 못 하는 일이 생기면 **반드시 "🙋 도와줘요 개발자!"** 라고 외쳐서 사용자에게 알려준다.

예시 상황:
- 브라우저에서 로그인 / OAuth 승인
- 비밀번호·API 키 신규 발급 (Supabase 대시보드, GitHub settings 등)
- 결제·요금제 변경
- 휴대폰 인증
- 로컬 환경에 뭔가 설치 (Node, Python 런타임 등)
- Supabase 대시보드에서 SQL 직접 실행 (선택지로 제시: "제가 SQL 드릴게요, 직접 실행하실래요? 아니면 개발자분께 부탁할까요?")

**외치는 법 예시:**
> 🙋 **도와줘요 개발자!**
> Supabase 대시보드 로그인은 제가 못 해요. 로그인 상태로 이 SQL을 실행해주셔야 해요.
> → 어렵다 싶으면 개발자분께 위 SQL 전달하시면 됩니다.

## 🧩 이 프로젝트 고정 규칙

### Soft delete 원칙
- **절대 `DELETE` SQL 쓰지 않는다.**
- "삭제"는 모두 `consumed_at` (소비) 또는 `deleted_at` (삭제) 타임스탬프 UPDATE로.
- 목록 조회는 `WHERE consumed_at IS NULL` 필터 필수.
- **이유:** 사용자가 나중에 "지난주에 뭐 먹었지?" 같은 이력 조회를 원할 수 있음.

### Supabase 사용법
- 클라이언트는 `lib/supabase.ts` 를 import해서만 사용. 다른 데서 `createClient` 직접 호출 금지.
- 환경변수: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`
- **`.env.local`은 절대 커밋하지 않는다.** (`.gitignore`에 포함됨)
- 새 키 추가 시 `.env.example`도 같이 업데이트.

### 모바일 퍼스트
- 레이아웃은 **폰 화면 기준으로 먼저** 설계.
- Tailwind 반응형은 `sm:` `md:` `lg:` 로 점진적 확장만 사용.
- 폰트 크기, 터치 영역(최소 44px) 고려.

### 문서 동기화 (중요)
- 새 기능 추가·변경 시 → `docs/requirements.md` 업데이트
- 기획 변경·의사결정은 → `docs/planning.md` "의사결정 로그"에 날짜와 함께 기록
- 스키마 변경은 → `docs/schema.md` 에 SQL 반영
- `README.md`는 **최종 정리본** (완성된 기능만 반영, 실험/미완성 기능 X)

## 🧯 에러/트러블슈팅 흐름

에러가 발생하면 이 순서로:

1. **에러 메시지를 한국어로 번역** + 무슨 뜻인지 풀어서 설명
2. **가능한 원인 2~3개** 나열 (가장 흔한 것부터)
3. **확인 방법** 제시 (사용자 부담 최소화)
4. **해결책 여러 안 + 추천안**
5. 해결 후 **"다음에 또 만나면 어떻게 알아볼지"** 한 줄 정리

## 📝 커밋 & 푸시 규칙
- **사용자 확인 없이 커밋/푸시 절대 금지.** 항상 "커밋할까요?" 물어보기.
- 커밋 메시지는 한국어 OK. "무엇을/왜"를 중심으로.
- `.env*` 는 커밋 금지. (`.env.example` 만 예외)
- 민감정보(비밀번호, 진짜 API 키) 발견 시 커밋 중단하고 경고.
