# sj-fresh 기획 문서

> 어떻게 만들지, 화면 흐름, 의사결정 로그.

## 기술 스택
- **프론트엔드**: Next.js 15 (App Router) + TypeScript + Tailwind CSS
- **백엔드**: 별도 서버 없음 — Supabase 직접 호출
- **DB**: Supabase (PostgreSQL)
- **파일 저장소**: Supabase Storage (재료 사진용)
- **배포**: Netlify (GitHub 연동 자동 배포)

## 화면 흐름 (MVP)

```
[메인 = 재료 목록]
  ├─ [+] 재료 추가 버튼 → [추가 화면]
  ├─ 각 재료 카드 탭 → [소비 처리]
  └─ [이력] 탭 → [소비 이력 화면]
```

### 화면별 요소
- **메인(목록)**: 유통기한 임박 순 정렬, 남은 일수 뱃지(D-3 같은 식), 카드 탭 → 소비 처리
- **추가 화면**: 이름, 넣은 날짜(기본 오늘), 유통기한, (옵션) 사진 첨부
- **이력 화면**: 소비한 재료 리스트, 언제 먹었는지 표시

## 의사결정 로그

### 2026-04-19
- **Python → Next.js 변경**: Netlify가 Python 서버를 제대로 지원 안 해서 스택 변경. Supabase가 백엔드 역할 다 해주므로 서버 따로 없어도 됨.
- **레포 구조**: `C:\sj`는 워크스페이스, 각 프로젝트는 독립 레포. sj-fresh도 독립 레포.
- **soft delete 방식 확정**: `deleted_at`보다 **`consumed_at`** 선호 — 왜 사라졌는지 의미가 더 명확.
